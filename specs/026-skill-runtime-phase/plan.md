# Implementation Plan: Generic Skill Runtime (Phase 2)

**Branch**: `026-skill-runtime` | **Date**: 2026-06-06 | **Spec**: `specs/026-skill-runtime-phase/spec.md`
**Input**: Feature specification from `/specs/026-skill-runtime-phase/spec.md`

## Summary

Activate the deferred generic skill runtime on top of Phase 1's builder-runs stack. Add a boot-time skill registry rooted at `$SKILLS_ROOT`, a frontmatter parser, an active-template scanner that reads `requiredSkills` / `recommendedSkills` from the four active prompt-template families, a hybrid detector (deterministic scoring + LLM tie-break for tight ambiguity), a clarification flow that pauses the builder run BEFORE creating a draft workspace, agentic-loop injection via the `<selected_skill>` wrapper, and a Codex tool `project_read_skill` for self-directed skill use. Land `design-taste-frontend` as the seeded skill at `$SKILLS_ROOT/design-taste-frontend/SKILL.md`. Every change is additive on top of the Phase 1 schema, lifecycle, and API; no Phase 1 surface gets renamed or removed.

## Technical Context

**Language/Version**: TypeScript on Node.js 22 LTS (per `docs/deploy-vps.md` and Phase 1 plan).
**Primary Dependencies**: `@openai/codex-sdk` (Phase 1), TanStack Start, Drizzle ORM, PM2, Vitest, pnpm 10. Phase 2 adds: `js-yaml` (already in dependencies ‚Äî confirmed in `package.json`) for SKILL.md frontmatter parsing.
**Storage**: PostgreSQL via Drizzle (`src/db/schema.ts`); on-disk skills under `$SKILLS_ROOT` (defaults: `/var/bin/skills` prod, `process.cwd()/skills` dev).
**Testing**: Vitest unit + integration. Mock the `@openai/codex-sdk` provider for tie-break tests. Mock filesystem for registry-boot tests via real `fs.mkdtemp` per Phase 1 convention.
**Target Platform**: Linux VPS (production) + macOS / Linux dev workstations.
**Project Type**: Web app (TanStack Start single project) with auxiliary worker.
**Performance Goals**: Skill registry boot ‚â§ 500ms p95 for 50 skills; deterministic detector ‚â§ 5ms per run; LLM tie-break adds ‚â§ 2s to ambiguous runs (skipped on the 95% non-ambiguous path).
**Constraints**: One global `$SKILLS_ROOT`; no hot-reload; tie-break uses bounded ‚â§ 1k input tokens per call; no raw SKILL.md body persisted in DB or logs (extension of Phase 1 FR-040).
**Scale/Scope**: Phase 2 ships with one seeded skill (`design-taste-frontend`); the runtime supports up to ~50 skills before re-evaluating registry caching. ‚â§ 100 active projects per VPS unchanged from Phase 1.

## Phased Implementation

Each phase ends with `pnpm typecheck && pnpm lint && pnpm test` green. Phases ship as separate PRs. Within this branch the phases collapse into one PR ‚Äî but commits stay phase-aligned.

### Phase 0 ‚Äî Sync (this PR header only)

- Spec + plan + clarify already locked. No grill artifacts pending.
- This phase produces no source-code changes; it captures the constitution check + locked decisions.

### Phase 1 ‚Äî Foundational: schema + registry + parser

- Drizzle migration: add `selectedSkills json()`, `pendingSkills json()`, `loadedSkills json()` to `builder_runs`. Defaults `[]`. Existing `selectedInstructions[]` and `pendingInstructions[]` columns stay.
- Wire `SKILLS_ROOT` into `src/server/env/codex.ts` so it is consumed (not just reserved). Default still `/var/bin/skills` prod, `process.cwd()/skills` dev. Missing dir ‚Üí empty registry, not a fatal error (FR-007).
- New `src/features/agents/codex/skills/frontmatter-parser.ts`: strict zod schema for `name`, `description`, `aliases[]`, `triggers[]`, `asksClarification`, `clarificationPolicy`, `appliesTo[]`. Reject unknown values. Strip frontmatter; return body + meta.
- New `skills/skill-loader.server.ts`: per-file read, truncate at `MAX_SKILL_CHARS` (env, default 32k), compute SHA-256 hash, return `{ meta, body, hash, truncated }`.
- New `skills/registry.server.ts`: boot-time scan of `$SKILLS_ROOT/*/SKILL.md`. Skip on parse failures, name mismatch, duplicate `name`, symlink escapes. Emit `skill_registry_loaded` and `skill_load_failed` audit events. Expose `getSkill(name)`, `listSkills()`, `getRegistryStatus()`.
- Unit tests for parser, loader (truncation + hash), registry boot resilience.

### Phase 2 ‚Äî Active-template scanner

- New `skills/template-scanner.server.ts`: scans only the 4 active prompt-template families (`templates/codex-builder/foundation/edit-system.md`, `templates/codex-builder/init/system.md`, `templates/codex-builder/recovery/*.md`, `templates/codex-builder/redesign/*.md`). Picks up YAML frontmatter `requiredSkills` / `recommendedSkills` and inline `@skill:<name> required|recommended` directives.
- Refuses to declare a skill that isn't in the registry (FR-009 ‚Äî caller surfaces the failure at runtime, not boot time).
- Unit tests covering frontmatter, inline directives, mixed directives, missing-file resilience.

### Phase 3 ‚Äî Detector + hybrid tie-break

- New `skills/detector.server.ts`: scoring matrix (template required +100, alias/explicit-mention +80, template recommended +60, trigger phrase +25, description keyword cluster +15, appliesTo +10) with thresholds (>=80 auto / 50‚Äì79 candidate / 30‚Äì49 metadata / <30 ignore). Cap at `MAX_SELECTED_SKILLS` (env, default 3) with required-overrides-cap.
- New `skills/tie-break.server.ts`: only invoked when two candidates fall in the 50‚Äì79 band with score gap ‚â§ `LLM_TIE_BREAK_GAP` (env, default 10). Uses Codex SDK provider config (existing `@openai/codex-sdk` instance) with structured output schema `{ pick: string | null, confidence: number, reason: string }`. Sends metadata only ‚Äî never SKILL.md body.
- New `skills/selection.server.ts`: orchestrator. Calls detector ‚Üí if tight ambiguity, call tie-break ‚Üí if tie-break fails / returns ambiguous, escalate to clarification.
- Unit tests for detector (table-driven across the matrix + thresholds), tie-break dispatch (mock LLM client returning `tie: true` / `pick: "x"` / network error ‚Üí escalation), selection orchestrator routing.

### Phase 4 ‚Äî Clarification flow

- Extend `builder-run.server.ts`: BEFORE creating draft workspace, run selection. If selection returns `awaiting_clarification`, persist `pendingSkills[]` to `builder_runs.metadata`, emit `awaiting_clarification` SSE event, and return without creating a draft.
- New `skills/clarification.server.ts`: builds product-safe question + 3‚Äì4 bounded options + free-text fallback from the candidate skills' descriptions.
- New `POST /api/projects/$projectId/builder-runs/$runId/answer`: validates non-empty answer, re-runs detector with `originalPrompt + answer + pendingSkills metadata`, removes the pause, continues the run from where it stopped.
- Reuse Phase 1 cancel path: cancel during `awaiting_clarification` ‚Üí terminates with `cancelled`, retains `pendingSkills[]` for audit.
- Integration test `skill-clarification-flow.test.ts`: tie ‚Üí paused ‚Üí no draft on disk ‚Üí answer ‚Üí resumed ‚Üí draft created at this point ‚Üí injected.
- Integration test `skill-required-missing.test.ts`: required missing ‚Üí fail-fast ‚Üí no draft on disk.
- Integration test `skill-tie-break-fallback.test.ts`: LLM provider down ‚Üí escalation to user clarification (not silent pick).

### Phase 5 ‚Äî Injection + Codex tool

- New `skills/injection.server.ts`: emits `<selected_skill name="..." version="..." hash="..." source="..." score="...">‚Ä¶body‚Ä¶</selected_skill>` blocks. Order: required first, then candidates by descending score, then polish slot. Plug into `context-builder.server.ts` next to the existing `<selected_instruction>` emitter so they coexist.
- New `skills/project-read-skill.tool.server.ts`: bounded tool surface. Resolves only via registry. Rejects names containing `..`, path separators, absolute paths. Returns structured error to Codex on miss / boundary attempt; appends to `loadedSkills[]` audit on success.
- Wire the tool into `codex-thread.server.ts` via the SDK config layer (Phase 1 builder-run wraps `codex-sdk` Thread; tool registers there).
- Unit tests for injection wrapper format + ordering, tool boundary (path traversal, missing skill, registry miss).

### Phase 6 ‚Äî Schema columns + run metadata

- Drizzle migration: column additions (already done in Phase 1 of this plan; this phase is the migration wiring + repository updates).
- Update run record save path to populate `selectedSkills[]`, `pendingSkills[]`, `loadedSkills[]`.
- Audit event emitters: `skill_registry_loaded`, `skill_selected`, `skill_load_failed`, `skill_injected` per FR-039.

### Phase 7 ‚Äî Seed skill + active template wiring

- Provision `skills/design-taste-frontend/SKILL.md` (dev default path) with full frontmatter (`name`, `aliases: [taste skill, anti-slop]`, `triggers: [redesign, premium UI, storefront UI]`, `asksClarification: true`, `clarificationPolicy: when_ambiguous`, `appliesTo: [init_project, design_update, ui_mutation]`) and design-taste body content.
- Update `templates/codex-builder/init/system.md` frontmatter to declare `requiredSkills: [design-taste-frontend]`. Verify via the template-scanner test that the new declaration is picked up.

### Phase 8 ‚Äî UI + locale

- Extend `BuilderRunProgress` with `awaiting_clarification` milestone label, plus an inline question card with the 3‚Äì4 option buttons + free-text input.

## Risks & Mitigations

- **`$SKILLS_ROOT` permission drift on VPS** ‚Üí Phase 1's `docs/deploy-vps.md` already pins ownership/mode (`builder:builder` 750). Phase 2's loader skips unreadable entries with a warning rather than failing boot, so a partial misconfig does not block the app (FR-007).
- **Tie-break LLM cost / latency creep** ‚Üí Detector logs whether tie-break fired per run. Selection bands (50‚Äì79 with ‚â§ 10 gap) are tunable env vars (`LLM_TIE_BREAK_GAP`). If telemetry shows tie-break firing on > 5% of runs (SC-006), tune the gap up or refine `triggers` in seed skills.
- **Provider down ‚Üí endless clarification loops** ‚Üí FR-014 escalates the ambiguity to user clarification when tie-break fails. UI surfaces a clarification card with cancel option (Phase 1 cancel path). Run is never silently picked.
- **`<selected_skill>` injection bloats Codex prompt** ‚Üí `MAX_SKILL_CHARS` (default 32k) + `MAX_SELECTED_SKILLS` (default 3) cap total injected content at ~96k chars. Required overrides allow exceeding the count cap; the char cap still applies per skill.
- **Two seed skills race on `name`** ‚Üí Loader keeps first-loaded; `skill_load_failed` with reason `duplicate_name` for the others. Phase 2 ships only one seed skill so the race surface is empty for now.
- **Audit DB bloat from `loadedSkills[]`** ‚Üí Field stores names only, not bodies. A run loading 50 skills via `project_read_skill` adds < 1KB to the row.
- **Clarification answer free-text injection attempt** ‚Üí Detector explicitly does NOT detect skills purely from the answer's free text (FR-021). Only re-scoring against the original prompt + the answer-as-context.
- **Phase 2 schema change conflicts with Phase 1 deferred drop of `agent_runs`** ‚Üí Phase 2 only ADDS columns to `builder_runs`. Phase 11 of `025-codex-sdk-migration` (agent_runs drop) is a separate migration on a separate timeline.

## Out of Scope (Phase 2)

- Rewriting the legacy AI Agent UI (`src/routes/projects/$projectId.tsx` and siblings).
- Deleting `src/features/ai-agent/*` or dropping `agent_runs`. Both belong to the Phase 1 follow-up PR.
- Multi-tenant skill registries / per-project overrides.
- Hot-reloading skills during a running app process.
- Skill marketplace / external skill download.
- Backward-compat aliases for the removed `tasteSkillLoaded` flag or `project_read_taste_skill` tool.
- Migrating existing Phase 1 foundation templates from `<selected_instruction>` into skills wholesale. That conversion is opt-in, one foundation block at a time, in follow-up PRs.
