# Implementation Plan: Codex SDK Migration

**Branch**: `025-codex-sdk-migration` | **Date**: 2026-06-05 | **Spec**: `specs/025-codex-sdk-migration/spec.md`

## Summary

Replace the in-house AI Agent runtime under `src/features/ai-agent/*` with `@openai/codex-sdk`-driven builder runs. App owns lifecycle, draft workspace, validation/preview/promote, boundary enforcement, retention, UI events. Codex SDK only writes inside an assigned draft workspace per run. Old run data and project state are dropped (pre-production, single developer). Retail templates are migrated wholesale into `templates/codex-builder/{foundation,init,recovery,redesign}/*` to keep retail know-how alive while removing the old runtime. Generic skill runtime is deferred but the lifecycle keeps stable extension points.

## Technical Context

**Language/Version**: TypeScript on Node.js 22 LTS (per `docs/deploy-vps.md`).
**Primary Dependencies**: `@openai/codex-sdk` (new), TanStack Start, Drizzle ORM, PM2 (process manager), Vitest (tests), pnpm 10.
**Storage**: PostgreSQL via Drizzle (`src/db/schema.ts`); on-disk draft + published workspaces under `/var/bin/projects/<projectId>/{draft,published}` in production and `process.cwd()/projects/<projectId>/...` in development.
**Testing**: Vitest unit + integration; Playwright optional for preview health smoke (out of phase 1 if not already wired).
**Target Platform**: Linux VPS (production), macOS / Linux dev workstations.
**Project Type**: Web app (TanStack Start single project) with auxiliary worker.
**Performance Goals**: Init end-to-end ≤ 6 min p95 on a clean project; small update ≤ 60s p95; cancel acknowledged ≤ 2s.
**Constraints**: One builder run per project at a time; no token budget enforcement in phase 1 (log only); no raw prompt/diff/full content in logs.
**Scale/Scope**: One VPS, port range 10000–19999 for previews, ≤ 100 active projects in phase 1.

## Phased Implementation

Each phase ends with `pnpm typecheck && pnpm lint && pnpm test` green. Phases ship as separate PRs.

### Phase 0 — Sync grill summaries + scaffold spec (this PR)

- Update `docs/codex-sdk-migration-grill-summary.md` and `skill-runtime-discussion-summary.md` to remove the BC alias text (Q1) and to reference `templates/codex-builder/*` as the unified instruction location (Q2/Q3/Q4).
- Land `specs/025-codex-sdk-migration/{spec,plan}.md`.
- Update `docs/deploy-vps.md` with Codex SDK + skills folder provisioning.

### Phase 1 — Templates migration

- `git mv` retail templates into `templates/codex-builder/{foundation,init/{data,pages},recovery,redesign}/*` per FR-003.
- Update every reference inside `src/features/ai-agent/*` to the new path; once Phase 5 deletes those files, the cleanup is complete.
- Verify `pnpm typecheck` passes (TypeScript may not catch path strings; spot-check via grep before/after).
- No behavioral change yet.

### Phase 2 — Foundation: env + Codex config + module skeleton

- Add `@openai/codex-sdk` dependency.
- New env vars (validate at startup, fail-soft → disable Codex feature on missing/invalid):
  - `CODEX_HOME` (path, defaults to `/var/bin/cloud-ai/codex-home` in prod, `process.cwd()/.cache/codex-home` in dev)
  - `CODEX_API_KEY`
  - `CODEX_MODEL`
  - `CODEX_BASE_URL` (optional; depends on provider)
  - `SKILLS_ROOT` (defined now for forward compat, not consumed in phase 1)
- Implement `src/features/agents/codex/runtime/codex-config.server.ts`: writes a profile file under `$CODEX_HOME/config.toml` with `env_key` references; deletes the file on app shutdown.
- Module skeleton folders + barrel exports.
- Feature flag `CODEX_FEATURE_AVAILABLE` derived from env validation; UI consumes it to render the unavailable banner.

### Phase 3 — Database schema + neutral infrastructure move

- Drizzle migration: drop `agent_runs` (and any auxiliary tables that belong only to the old agent), add `builder_runs` with columns from FR-036. JSON fields use `json()` per Principle IX.
- Reset old `project_state` columns that the new schema replaces (preserve only project identity, owner, slug, preview port allocation).
- Move neutral infra modules from `src/features/ai-agent/*` into `src/features/{projects,runtime,generated-projects}/*` with `git mv`. No logic changes in this phase — only path and import alias updates (Principle X).
- Delete clearly-unused agent code paths in the same PR if and only if they have no remaining importers.

### Phase 4 — Codex builder runtime

- `runtime/codex-thread.server.ts`: thin wrapper over Codex SDK creating a thread with `cwd = draftWorkspacePath`, `sandbox = workspace-write`, `roots = [draft]`, no `add-dir`.
- `runtime/builder-run.server.ts`: orchestrates one run end-to-end: load context → planning (init / new-route only) → mutate → validate → repair (≤ 2) → promote → publish.
- `runtime/repair-loop.server.ts`: re-feeds validation summary into the same thread; max 2 cycles.
- `runtime/cancel-controller.server.ts`: AbortController per run, called from the cancel API endpoint and from boundary-violation/timeout paths.
- `context/context-builder.server.ts`: loads templates from `templates/codex-builder/*`, project summary, file manifest, locale, and assembles the context bundle. Wraps instructions in `<selected_instruction name=… source="template_required">…</selected_instruction>` (FR-044).
- `context/project-summary.server.ts`: read latest approved summary; record version on the run.
- `context/instruction-loader.server.ts`: reads instruction files from disk; computes `hash`; returns `{name, source, version, hash, loaded:true}` metadata for `selectedInstructions[]`.
- `events/milestone-mapper.server.ts`: maps Codex SDK events → product-safe milestones; never streams raw events.
- Unit tests for context builder + milestone mapper.

### Phase 5 — Boundary, validation, retention, gates

- `boundary/protected-paths.ts`: blocked + allowed-with-audit lists from FR-020.
- `boundary/path-guard.server.ts`: validates `(projectId, userId, draftWorkspacePath)` for every run, every event with a path.
- `boundary/symlink-check.server.ts`: pre-run scan, fails closed.
- `boundary/filesystem-audit.server.ts`: snapshot before/after run + before promote.
- `boundary/diff-gate.server.ts`: post-run check that all changed paths fall under draft and respect protected-path policy.
- `boundary/promotion-gate.server.ts`: final guard before sync.
- `validation/typecheck.server.ts`, `build.server.ts`: spawn `pnpm` in draft workspace; capture stdout/stderr summary; never echo full output to UI.
- `validation/preview-health.server.ts`: HTTP probes against PM2 instance; derives URL list from manifest + new routes.
- `validation/pm2-status.server.ts`: thin wrapper over `pm2 jlist` parsed JSON.
- `validation/product-sample-parser.server.ts`: AST-based static evaluator for `productsListSample`; returns id + store-slug fallback chain (FR-031, FR-032).
- `retention/draft-retention.server.ts`: scheduled GC at 12h boundary; cancelled / failed / boundary drafts purged exactly once; promoted drafts deleted immediately post-promote.
- All boundary modules have unit tests with adversarial cases (path traversal, symlink escape, protected file write, etc.).

### Phase 6 — API + UI rewrite around builder-runs

- API routes under `src/routes/api/projects/$projectId/builder-runs/*` per FR-038.
- SSE stream emits product-safe milestones only; closes on terminal status.
- Cancel endpoint signals the run's AbortController; idempotent.
- Retry endpoint creates a fresh run; never reuses a draft.
- UI (existing `/projects` master-detail): replace agent-run progress with builder-run progress component consuming the new event union; failure code surface uses i18n strings (Vietnamese primary, English fallback).
- Unavailable banner triggered by `CODEX_FEATURE_AVAILABLE=false`.
- Concurrency rejection per FR-041 with friendly message.

### Phase 7 — Cleanup

- Delete `src/features/ai-agent/agent`, `src/features/ai-agent/code-tools`, `src/features/ai-agent/openai`, `src/features/ai-agent/planning`, `src/features/ai-agent/runtime`, `src/features/ai-agent/store-runtime`, `src/features/ai-agent/thinking`, `taste-skill-loader.server.ts`, `taste-skill-preload.server.ts`.
- Delete legacy taste-skill flag handling and `project_read_taste_skill`-related code.
- Remove now-unused fields from project state if their last reader is gone.
- `pnpm lint --fix` for stale imports; `pnpm typecheck` to verify nothing dangles.

### Phase 8 — Verification

- Manual end-to-end smoke against a dev VPS staging:
  1. Init prompt → published storefront with 5 core routes 200.
  2. Small/copy update → typecheck-only, no build.
  3. New-route update → planning + build + preview health.
  4. Failing draft → 2 repair cycles → success or `repair_exhausted`.
  5. Inject a deliberate path traversal in a fixture prompt → `boundary_violation` fail-closed.
  6. Cancel mid-run → `cancelled`, draft kept 12h.
  7. Remove `CODEX_API_KEY` → app starts, builder UI shows unavailable banner, other pages load.
- Sign off Phase 1 success criteria SC-001 → SC-010.

## Risks & Mitigations

- **Codex SDK API surface unstable** → wrap behind `runtime/codex-thread.server.ts`; only that file imports `@openai/codex-sdk`. Bump path is a single-file change.
- **Boundary enforcement gaps** → defense in depth (FR-016 lists 9 distinct layers); each has unit tests with adversarial inputs.
- **Template migration breaks dev-agent / other consumers** → `git mv` preserves history; spot-grep for old template paths before merging Phase 1; pre-production environment makes recovery easy.
- **Retention scheduler missing a tick** → idempotent GC keyed on `(runId, retainUntil)`; dedicated cron-style worker checked at app startup and every 30 min.
- **PM2 status flake** → wrap probes with single retry + 2s backoff; treat `not online` after retry as `preview_failed`, not generic error.
- **Forward-compat for skill runtime** → wrapper element + `selectedInstructions[] / pendingInstructions[]` fields persist now; migration to `selectedSkills` / `pendingSkills` is additive.

## Out of Scope (Phase 1)

- Generic skill runtime (`SKILLS_ROOT` consumption, skill detector, scoring, clarification) — design preserved in grill summary.
- Run queueing / parallel runs per project.
- Token budgeting / cost ceilings (log only).
- Full Zod / commerce contract validation; phase 1 reports `commerceValidationStatus: skipped`.
- Multi-tenant Codex profile pools.
- Cross-project run history dashboards.
