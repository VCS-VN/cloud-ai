# Feature Specification: Codex SDK Migration (Big-Bang Rewrite of AI Agent Builder)

**Feature Branch**: `025-codex-sdk-migration`
**Created**: 2026-06-05
**Status**: Draft
**Input**: Grill summaries `docs/codex-sdk-migration-grill-summary.md` + `skill-runtime-discussion-summary.md`, reconciled in this session.

## Context

The current custom AI Agent runtime under `src/features/ai-agent/*` is too inefficient and too expensive to maintain. This feature replaces it with `@openai/codex-sdk` as the coding engine for every project run (init / update / repair). The app keeps ownership of project lifecycle, context summarization, draft workspace, validation, preview health, promotion, and boundary enforcement; Codex SDK only writes inside an assigned draft workspace.

Phase 1 ships the migration without the generic skill runtime. The lifecycle is designed so the skill runtime can be activated in a later phase without changing the core flow. The existing retail templates under `templates/{agent-system,init-prompt,init-recovery,redesign}/*` are migrated into `templates/codex-builder/{foundation,init,recovery,redesign}/*` and become the source of foundation instructions for Codex builder runs.

## No Backward Compatibility

- Old run history (`agent_runs` and related tables) is dropped; no migration of historical runs.
- Old project state schema is reset; new builder flow starts from a fresh `builder_runs` schema.
- Existing project files on disk under `/var/bin/projects/*` are kept, but the app does not guarantee compatibility with them.
- `tasteSkillLoaded` flag and `project_read_taste_skill` tool are removed entirely (no alias/wrapper retained).
- `.agents/skills/*` and the hardcoded taste-skill loader/preload pipeline are removed.

## User Scenarios & Testing *(mandatory)*

### User Story 1 — Init a new retail storefront (Priority: P1)

A user creates a new project, types a natural-language prompt, and gets a working retail storefront with home / products / product-detail / cart / checkout / orders pages, generated by Codex SDK and published to the project's preview URL.

**Why this priority**: Init is the entry-point for every project. Without it, no other flow has anything to update.

**Independent Test**: From an empty project workspace, submit one init prompt; observe a builder run that goes through `loading context → planning → creating draft → building pages → checking preview → publishing → done`, and verify the preview URL serves the five core routes with HTTP 200.

**Acceptance Scenarios**:

1. **Given** a new empty project, **When** the user submits an init prompt, **Then** the system runs a single Codex thread with multiple batched turns (foundation/data → page batches → polish → final validate) and promotes only after all gates pass.
2. **Given** the init plan would touch a blocked path (e.g. `package.json`, `vite.config.ts`), **When** Codex proposes the plan, **Then** the system asks Codex to revise once; if still invalid, the run fails with `blocked_request` before any draft mutation.
3. **Given** a successful init promotion, **When** the run completes, **Then** the published workspace contains a valid `productsListSample` export and the preview PM2 instance serves `/`, `/products`, `/products/:sampleProductId`, `/cart`, `/checkout`.

---

### User Story 2 — Small/direct content or UI update (Priority: P1)

A user with an existing project asks for content / copy / sample-data changes or a UI tweak inside an existing route, and the change is published without a planning turn.

**Why this priority**: Small updates are the highest-frequency builder action; latency must stay low.

**Independent Test**: On a promoted project, submit a small-update prompt; observe the run skipping the planning turn, running typecheck + preview health (no full build), and promoting the draft.

**Acceptance Scenarios**:

1. **Given** an active project, **When** the user submits a content/copy or in-route UI change, **Then** the run skips planning, performs Codex turn directly, runs typecheck + preview health, and promotes if both pass.
2. **Given** a small update touches > 20 changed files, **When** the diff gate runs, **Then** the run is rejected with `validation_failed` and not promoted.

---

### User Story 3 — Add a new route / page (Priority: P2)

A user requests adding a new route or page (e.g. `/about`, `/blog`). The run requires a planning turn and a full build before promotion.

**Acceptance Scenarios**:

1. **Given** an active project, **When** the user requests a new route, **Then** the run runs a planning turn first, mutates the draft only after the plan is accepted, runs `pnpm build` plus preview health, and promotes only if all gates pass.
2. **Given** the new route is added, **When** preview health runs, **Then** the new route URL is included alongside the core route hard gate.

---

### User Story 4 — Validation failure with same-thread repair (Priority: P2)

A run produces draft files that fail typecheck/build/preview. The system feeds the validation summary back to Codex in the same thread and lets it attempt repair up to 2 cycles.

**Acceptance Scenarios**:

1. **Given** a draft fails typecheck, **When** repair cycle 1 runs, **Then** Codex receives the error summary in the same thread and may modify only files inside the draft workspace.
2. **Given** repair cycles 1 and 2 both fail, **When** cycle 3 would start, **Then** the run is marked `repair_exhausted` and not promoted.

---

### User Story 5 — Boundary violation fails closed (Priority: P1)

If Codex attempts to read or write outside the assigned draft workspace (or the diff gate detects a write outside it, or symlink escape is detected), the run fails immediately and never promotes.

**Acceptance Scenarios**:

1. **Given** a run in progress, **When** Codex emits a tool error or sandbox denial referencing a path outside the draft, **Then** the run is failed with `boundary_violation`, no repair is attempted, and the draft is retained 12h for restricted/internal review only.
2. **Given** the post-run diff gate, **When** it sees any changed file outside the draft workspace, **Then** the run is failed with `boundary_violation` before promotion.
3. **Given** repeated boundary violations on a single project, **When** the threshold is exceeded, **Then** the project is suspended pending review.

---

### User Story 6 — Cancellation (Priority: P2)

A user cancels an in-progress run; the active Codex turn is interrupted, no draft is promoted, and the user sees a clear "cancelled" message.

**Acceptance Scenarios**:

1. **Given** a run in `creating draft` or `building pages`, **When** the user cancels, **Then** the active Codex turn is interrupted, the run state becomes `cancelled`, and the draft is kept 12h for debug.
2. **Given** a cancelled run, **When** the user opens the project, **Then** the published workspace is unchanged.

---

### User Story 7 — Codex feature unavailable at startup (Priority: P3)

If required Codex env (API key, provider config) is missing or invalid at startup, the Codex Agent feature is disabled while the rest of the app keeps running. The UI shows a friendly unavailable message.

**Acceptance Scenarios**:

1. **Given** missing Codex env on app start, **When** a user opens the builder, **Then** the prompt input is disabled and the UI shows the unavailable message (English + Vietnamese variant configured per locale).
2. **Given** missing Codex env, **When** the user navigates to non-builder pages, **Then** those pages still load.

---

### User Story 8 — Concurrency rejection (Priority: P3)

Phase 1 runs at most one builder run per project at a time.

**Acceptance Scenarios**:

1. **Given** a project with an active run, **When** the user submits a new prompt, **Then** the system rejects with a friendly message; no queue is created.

---

### Edge Cases

- A user prompt requires editing a blocked path → run does not start a Codex SDK call; user gets a friendly warning suggesting in-scope alternatives.
- An init batch plan returns more than 40 changed files → batch is split or reduced; runs over the limit are rejected before mutation.
- Product sample parser smoke check fails (export missing, non-JSON-compatible expression, function calls in data) → run fails with `validation_failed`; no commerce-correctness claim shown to user.
- Preview optional route returns non-200 → run still promotes; metadata records the optional failure; user-facing summary includes a soft warning.
- Symlink inside draft points outside draft → boundary check fails closed before Codex starts.
- Codex emits tokens but produces no diff → run fails with `validation_failed` ("no changes produced").
- Codex SDK process throws or exits unexpectedly → run fails with `codex_runtime_failed`; draft retained 12h.

## Requirements *(mandatory)*

### Functional Requirements

#### Module layout & ownership

- **FR-001**: System MUST place the new Codex builder code under `src/features/agents/codex/*`. Neutral product infrastructure MUST live under `src/features/projects/*`, `src/features/runtime/*`, `src/features/generated-projects/*`, and shared UI/event types under `src/features/agents/ui/*`.
- **FR-002**: System MUST delete the AI runtime internals under `src/features/ai-agent/agent`, `src/features/ai-agent/code-tools`, `src/features/ai-agent/openai`, `src/features/ai-agent/planning`, `src/features/ai-agent/runtime`, `src/features/ai-agent/store-runtime`, `src/features/ai-agent/thinking`, and the taste-skill loader/preload modules. Neutral pieces still required (project lifecycle, runtime, preview, retention) MUST be moved to the neutral paths above rather than deleted.

#### Templates migration

- **FR-003**: System MUST migrate retail instruction templates from their current locations into `templates/codex-builder/{foundation,init,recovery,redesign}/*` using `git mv` to preserve history. The new layout is:
  - `templates/codex-builder/foundation/{retail-constraints,reasoning-workflow,edit-system}.md`
  - `templates/codex-builder/init/{system,init-mode,manifest.json}` plus `init/data/{catalog-data,data,packages,provider,component}.md` and `init/pages/{home,products,product-detail,cart,checkout,orders,order-detail}.md`
  - `templates/codex-builder/recovery/{recovery,server-design-guidance,vertical-guidance}.md`
  - `templates/codex-builder/redesign/{anti-slop-repair,redesign-rewrite,token-patch-rewrite}.md`
- **FR-004**: All references to old template paths inside `src/features/ai-agent/*` are removed together with the runtime; the new Codex context builder MUST read instructions only from `templates/codex-builder/*`.

#### Codex SDK runtime

- **FR-005**: System MUST run `@openai/codex-sdk` in-process inside the backend / app worker process. No separate Codex app-server subprocess in phase 1.
- **FR-006**: For each builder run the system MUST create a fresh Codex thread, set `cwd` to the assigned draft workspace, set sandbox to `workspace-write`, and never call `add-dir` or pass extra roots.

#### Codex config management

- **FR-007**: On startup the app MUST validate required Codex env (API key, model, base URL if needed). On validation failure the app MUST disable the Codex Agent feature and serve the unavailable message; the rest of the app keeps running.
- **FR-008**: The app MUST own a single global `CODEX_HOME` directory (not inside any project workspace) and MUST generate the Codex profile config at runtime, with secrets supplied via custom-provider `env_key` references rather than literal values in the file.
- **FR-009**: The app MUST NOT silently fall back to a different provider/model if Codex is unavailable.

#### Project context model

- **FR-010**: Each builder run MUST inject a compact project context bundle: latest approved project summary, file manifest/tree, the user prompt, locale rule, draft workspace path, validation/promote rules, protected-path policy, and the rendered foundation instructions block.
- **FR-011**: System MUST maintain an authoritative project summary versioned per project. Routine summaries auto-approve; identity- or commerce-impacting summaries require explicit approval before they become the active summary.
- **FR-012**: Phase 1 MUST NOT enforce a token budget; if Codex SDK exposes token counts, those counts MUST be logged to internal audit metadata.

#### Draft workspace & promotion

- **FR-013**: System MUST create one draft workspace per builder run, isolated per project/user. Codex SDK MUST be the only writer of the draft.
- **FR-014**: Promotion MUST happen only after all gates pass (diff gate → typecheck → build (when required) → preview health → core route check → product-sample parser smoke check). Promotion MUST sync the draft into the published workspace and then delete the full draft copy, while keeping diff/metadata artifacts per the project history retention policy.
- **FR-015**: Retention rules MUST be:
  - Cancelled / failed-validation drafts: keep 12h.
  - Boundary-violation drafts: keep 12h, restricted/internal access only.
  - Successful promoted drafts: full draft copy deleted post-promote; diff/metadata kept per retention.
  - User/admin delete request: removes retained draft immediately.

#### Boundary enforcement

- **FR-016**: System MUST enforce multi-layer boundary checks for every run: (a) prompt instruction, (b) `cwd` set to draft, (c) `workspace-write` sandbox, (d) no extra roots, (e) app-side path guard against `(projectId, userId, draftWorkspacePath)`, (f) symlink-escape check before run, (g) filesystem audit before/after run and before promote, (h) post-run diff gate, (i) promotion gate keyed on the same `(projectId, userId)`.
- **FR-017**: On boundary violation the run MUST be failed immediately, MUST NOT retry/repair/promote, MUST be tagged internally `BOUNDARY_VIOLATION`, and MUST surface only a product-safe message to the user.
- **FR-018**: Repeated boundary violations on a single project MUST trigger project-level suspension pending review; repeated violations across projects of one user/org MUST escalate.
- **FR-019**: Logs and user-visible events MUST NOT leak absolute paths or content from other projects.

#### Protected path policy

- **FR-020**: System MUST enforce a mixed protected-path policy with the blocked list: `package.json`, `pnpm-lock.yaml`, `package-lock.json`, `yarn.lock`, `bun.lockb`, `.env`, `.env.*`, `src/routes/__root.tsx`, `src/main.*`, `src/router.*`, `vite.config.*`, `tsconfig.json`, `tailwind.config.*`, `postcss.config.cjs`, `public/sw.js`. Allowed-with-audit paths: `src/routes/**`, `src/components/**`, `src/styles/**`, `src/lib/**`, `src/hooks/**`, `src/shared/sample-data/**`, `public/assets/**`.
- **FR-021**: A request requiring a blocked path MUST NOT trigger a Codex SDK call or draft mutation; the user MUST receive a friendly warning proposing in-scope alternatives, and the blocked intent MUST be logged internally.

#### Changed-file & batching limits

- **FR-022**: Normal updates MUST cap changed files at 20 per run.
- **FR-023**: Init runs MUST cap changed files at 40 per batch and use hybrid batching: foundation/data → page/route batches → polish → final validation. The whole init run MUST share one Codex thread across batched turns.
- **FR-024**: Init MUST require a planning turn before any mutation. Adding a new route/page MUST also require a planning turn. Small/direct updates MUST run without a planning turn.

#### Validation & repair

- **FR-025**: Phase 1 validation gates MUST include `pnpm run typecheck` (always), `pnpm run build` (init + new route/page updates), preview health check, PM2 instance/status check, and core-route HTTP checks.
- **FR-026**: Small/direct updates MAY skip the build step if typecheck and preview health pass.
- **FR-027**: Repair MUST happen in the same Codex thread, MUST cap at 2 repair cycles, and MUST mark the run failed if cycle 2 still fails.
- **FR-028**: System MUST use the failure taxonomy: `validation_failed`, `boundary_violation`, `config_unavailable`, `cancelled`, `preview_failed`, `codex_runtime_failed`, `blocked_request`, `repair_exhausted`.

#### Preview health

- **FR-029**: Preview health MUST verify the PM2 instance exists and is healthy, the root URL returns 200, and the core hard-gate routes (`/`, `/products`, `/products/:sampleProductId`, `/cart`, `/checkout`) return 200. New routes from the current run MUST be added to the URL list.
- **FR-030**: An optional-route 200 failure MUST NOT block promotion but MUST be recorded in metadata and surfaced as a soft warning in the user-facing summary.
- **FR-031**: The product-detail sample id MUST be derived from the parsed `productsListSample.data[0]` (id → entityId → defaultModel.productId), not from Codex text output.

#### Product sample data contract

- **FR-032**: `src/shared/sample-data/products.ts` MUST export `productsListSample` with the raw API product list response shape, at minimum `{ total: number, data: Product[] }`. Codex MAY update values inside the shape but MUST NOT add, rename, remove, or reshape fields.
- **FR-033**: System MUST parse `products.ts` with an AST/static evaluator only; the file MUST be JSON-compatible (no function calls, no runtime imports, no complex computed expressions). Phase 1 MUST run a parser smoke check that extracts product id and store-slug fallback; phase 1 MUST NOT claim full commerce validation.
- **FR-034**: If project-detail `.env` lacks `VITE_STORE_SLUG`, the runtime MUST fall back via `productsListSample.data[0].store.slug` then product id; the app MUST NOT write `VITE_STORE_SLUG` automatically.
- **FR-035**: Sample images MUST use seeded Picsum URLs (`https://picsum.photos/seed/<stable-seed>/<w>/<h>`); allowlist `https://picsum.photos/seed/` for phase 1.

#### Builder-run schema, API, UI events

- **FR-036**: System MUST create a `builder_runs` table with at minimum: `id`, `projectId`, `userId`, `kind` (init / update / new_route / repair), `status`, `failureCode`, `startedAt`, `completedAt`, `draftWorkspacePath`, `selectedInstructions[]` (metadata only — `name`, `source`, `version`, `hash`, `loaded`), `pendingInstructions[]` (forward-compat for clarification), `commerceValidationStatus`, and audit metadata (changed file paths, validation result, error code, config/model/provider, context summary version, token usage if available).
- **FR-037**: System MUST drop / ignore the legacy `agent_runs` table. Old runs MUST NOT be migrated.
- **FR-038**: API MUST expose builder-run routes: `POST /api/projects/:projectId/builder-runs`, `GET /api/projects/:projectId/builder-runs/:runId/stream`, `POST /api/projects/:projectId/builder-runs/:runId/cancel`, `POST /api/projects/:projectId/builder-runs/:runId/retry`.
- **FR-039**: UI MUST consume only product-safe milestone events: `loading_context`, `planning`, `creating_draft`, `building_pages`, `checking_preview`, `repairing`, `publishing`, `done`, `failed`, `cancelled`. Raw Codex events MUST NOT reach user UI.
- **FR-040**: Logs / events MUST NOT include the raw user prompt, the raw diff, or full file content. Internal audit metadata MUST be sufficient to debug without storing those.

#### Concurrency & data reset

- **FR-041**: Phase 1 MUST allow at most one active builder run per project; new prompts during an active run are rejected with a friendly message and MUST NOT be queued.
- **FR-042**: On rollout, old project state MUST be reset to the new schema; existing on-disk project files MAY remain but are not guaranteed compatible.

#### Skill-runtime forward compatibility (no implementation in phase 1)

- **FR-043**: The Codex builder lifecycle MUST keep these slots in order: load project context → detect/select instruction sources → resolve clarification before mutation → build Codex context bundle → run Codex SDK in draft workspace → validate/repair/promote.
- **FR-044**: Phase 1 instruction injection MUST use a stable wrapper compatible with future skill injection: `<selected_instruction name="..." source="template_required">…</selected_instruction>`. The future variant `<selected_skill …>` MUST be a drop-in replacement at the same slot.
- **FR-045**: Builder-run state MUST be able to store `pendingInstructions[]` and an extension field for `pendingSkills[]` later. Runs awaiting clarification MUST NOT create or mutate a draft workspace.
- **FR-046**: System MUST NOT include `tasteSkillLoaded`, `project_read_taste_skill`, or any backward-compat alias to the removed taste-skill behavior.
- **FR-047**: `SKILLS_ROOT` and the generic skill runtime are deferred; phase 1 MUST NOT load `$SKILLS_ROOT/*/SKILL.md`. The deploy doc MUST still describe how to provision the directory so phase 2 enablement is friction-free.

#### Cancellation

- **FR-048**: On user cancel the system MUST interrupt the active Codex turn, mark the run `cancelled`, retain the draft for 12h, and show the user a "no changes were published" message.

### Key Entities

- **BuilderRun**: One Codex-driven attempt to init/update a project. Owns lifecycle status, kind, draft workspace path, gate results, selected instruction metadata, audit metadata, failure code.
- **DraftWorkspace**: Per-run isolated copy of the project source where Codex SDK writes. Lifecycle ends on promote, cancel, fail, or retention timeout.
- **PublishedWorkspace**: Authoritative project source served by PM2 preview. Updated only via successful promote.
- **ProjectSummary**: Versioned authoritative summary used as Codex context. Distinguishes routine (auto-approved) and identity/commerce-impacting (approval required) versions.
- **SelectedInstruction (phase 1)**: Metadata for a foundation template injected into a run (`name`, `source`, `version`, `hash`, `loaded`). No raw content stored.
- **PendingInstruction**: Placeholder forward-compat record for an instruction whose clarification is unresolved. Phase 1 may stay empty; phase 2 fills with skill clarification entries.
- **ProductsListSample**: Authoritative sample data file at `src/shared/sample-data/products.ts`. Shape must match raw API response; values may change.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: Init flow produces a published storefront whose 5 core routes (`/`, `/products`, `/products/:sampleProductId`, `/cart`, `/checkout`) return HTTP 200 from the preview PM2 instance, in a single end-to-end run.
- **SC-002**: Small/direct updates promote without running `pnpm build`; only typecheck + preview health are gating.
- **SC-003**: New-route updates run a planning turn and a full build before promotion.
- **SC-004**: A draft that fails typecheck recovers via at most 2 repair cycles in the same Codex thread; cycle 3 is impossible.
- **SC-005**: Any boundary violation (Codex tool error, sandbox denial, app filesystem audit, diff gate, promotion gate) fails the run with `boundary_violation`; no repair, no promote.
- **SC-006**: Cancelled / failed / boundary drafts are retained exactly 12h then garbage-collected; promoted drafts have no full copy retained.
- **SC-007**: Old `agent_runs` data is removed; the app starts and serves builder-runs against an empty `builder_runs` table on first deploy.
- **SC-008**: With Codex env removed, the app starts, non-builder pages load, and the builder UI shows the localized unavailable message.
- **SC-009**: Logs / events / persisted metadata contain no raw prompt, no raw diff, no full file content; observers can debug a run from metadata alone.
- **SC-010**: Phase 2 skill-runtime activation can be added by replacing `<selected_instruction>` with `<selected_skill>` and populating `pendingSkills[]` without touching the lifecycle order.

## Assumptions

- The VPS already has Node 22, pnpm, PM2, Cloudflared, and the directories described in `docs/deploy-vps.md`.
- A valid `@openai/codex-sdk`-compatible API key and model are available via env on the deploy host.
- The project is pre-production and one-developer; destructive schema/state resets are acceptable (per project memory).
- The builder app and preview router on the same VPS continue to serve traffic via the existing Cloudflare Tunnels.
- Locale is fixed to Vietnamese-primary for user-facing strings, with English equivalents for internal logs and admin messages.
- The generic skill runtime is intentionally out of scope; its design constraints only shape extension points, not phase-1 behavior.
